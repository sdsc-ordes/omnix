01 The big picture

What omnix is, in one breath — and the single design decision that shapes everything else.

omnix extracts breast-cancer xenograft sample data from the lab's SLIMS system and serves it as a web app you can browse and filter. That's it. The whole codebase is four ideas: talk to SLIMS, reshape the data, store it locally, serve a UI over it.

The one decision to internalise: there is a hard boundary in the middle. Getting data out of SLIMS is slow (≈22 000 records) and gated (needs the EPFL VPN). Browsing needs to be fast and always-available. So we split the work into two commands that meet at a file on disk:

02 SLIMS: the source of truth

Where the data lives, and why its shape is so unusual. This is the part most collaborators find confusing — so it's worth really getting.

SLIMS is a commercial Lab Information Management System. The UPBRI lab records every physical sample in it. Two surprises drive our whole approach:

Surprise 1: everything is in one table

Mice, tumors, blood vials, tissue for RNA — they're all rows in a single Content table. What kind of thing a row is comes from a foreign key, cntn_fk_contentType (Mouse = 19, Tumor = 10, Blood Sample = 21, …). To “fetch the mice” you query Content where cntn_fk_contentType = 19.

Surprise 2: cryptic column names, but they're systematic

Columns look like cntn_cf_mammoid or cntn_fk_status. There's a grammar: <4-letter table prefix>_<kind>_<field>. Decode a few yourself:

Every column carries four things

When you read a column from a record, you get a small bundle. This is why the code keeps reaching for displayValue:

AttributeExampleWhat it's for
namecntn_fk_statusthe cryptic machine id
title“Status”human label from the SLIMS UI
value10the raw value — for an FK, a bare primary key
displayValue“Available”the resolved label for that FK
Why it mattersFor a foreign key, value is a useless number (10) but displayValue is the meaningful label (“Available”). Our transform layer prefers displayValue and only falls back to resolving the pk itself.

Bonus: SLIMS even describes its own schema — there's a Field metadata table listing all 470 fields across 46 tables. That's how the docs/xenograft-data-model.md mapping and the UML diagram were built, rather than by guessing.

03 The domain model

The clean shapes we reshape SLIMS into — and the one join key that both connects everything and causes the most trouble.

Raw Content rows are painful to work with, so models.py defines three tidy pydantic entity models:

ModelFrom content typeKey fields
Tumor10slims_id, mammoid, tumor_type, HER2/ER/PR/Ki67, + derived flags
Mouse19slims_id, mammoid, generation, treatment, mutations_raw
Assay21, 5, …slims_id, assay_type, mammoid, organ

The join key: cntn_cf_mammoid

All three entities carry a mammoid field (things like T-050, M-37). It's the thread that links a tumor to its mice and assays. But it's clean only on tumors. On mice it's free text — you'll see 850-9 MCF7 shHR + Treatments as often as a tidy T-050.

The honest limitation (a great colleague question)On the real data, an exact mammoid match links only ~14% of mice to a tumor. That's mostly real biology, not a bug: most of those mice are cell-line models (MCF7), not patient-derived tumors, so they have no T-xxx to link to. There is no clean mouse→tumor foreign key in SLIMS.

Derived tumor fields

Some fields aren't stored anywhere — we compute them by looking at a tumor's linked mice and assays:

  • rna_sequenced — is there a linked “Tissue for RNA” assay?
  • dna_sequenced — is there a linked “DNA” sample? (always false today — there are zero DNA records in SLIMS)
  • n_experiments — count of distinct experiment numbers across linked samples
  • treatments — the unique treatments of the linked mice

05 The web app

Flask + HTMX + Jinja, no build step. How a filter keystroke becomes an updated table.

web/app.py is an app factory: create_app(db_path) builds a Flask app that opens the SQLite file read-only, once per request. The routes:

RouteDoes
/dashboard — counts + summary breakdowns
/<slug>full list page for tumors / mice / assays (filters + table)
/<slug>/rowsHTMX partial — just the table body, for live filtering
/tumor/<id>detail + drill-down into linked mice & assays
/export/<slug>.csv|.jsondownload the currently filtered set

The HTMX pattern — the one trick to understand

There's no JavaScript framework and no API. The filter form has HTMX attributes that say: “when an input changes, GET /tumors/rows and swap the result into #rows.” The server renders the same partial template it used for the initial page. Progressive enhancement: with JS off, the form still submits as a normal GET.

list.html — the filter form
<form hx-get="/tumors/rows"
      hx-target="#rows"
      hx-trigger="change, keyup delay:350ms">
   <!-- filter selects + inputs -->
</form>
<div id="rows"> <!-- _rows.html renders here, replaced on each change --> </div>
Why this is nice hereSingle-user, local, data-browsing tool → a full React SPA would be overkill. Server-rendered HTML + HTMX means no build step, no Node, and the templates are the whole UI. Flask was already a transitive dependency, so it cost nothing to adopt.

Security detail worth knowing

Filters build SQL WHERE clauses, so store.py keeps a whitelist (FILTER_COLUMNS) of which columns may be filtered. Values are always passed as bound parameters (?), never string-formatted. A malicious ?er=90; DROP TABLE tumor is simply ignored because er isn't whitelisted — there's a test for exactly that.

06 The database

One SQLite file, five real tables, a full-text index.

The snapshot lives at .omnix/snapshot.db (gitignored). Every entity table has a primary-key slims_id, the modelled columns broken out for filtering, and a raw_json column preserving the complete SLIMS record so nothing is lost.

store.py — SCHEMA (abridged)
tumor(slims_id PK, mammoid, tumor_type, subtype, grade, her2, er, pr, ki67,
      rna_sequenced, dna_sequenced, n_experiments, treatments, raw_json)
mouse(slims_id PK, mammoid, mouse_exp_nb, generation, treatment,
      mutations_raw, strain, sex, project, raw_json)
assay(slims_id PK, assay_type, mammoid, mouse_exp_nb, organ, raw_json)
snapshot_meta(created_at, source_url, counts_json)
  • Why SQLite? Zero-config, single file, no server. Perfect for a local single-user snapshot. It's stdlib — no extra dependency.
  • Indexes on mammoid (all three tables) plus type/generation columns keep filtering snappy at ~22k rows.
  • It's just a file — open it with sqlite3 .omnix/snapshot.db, DB Browser, or pandas.

07 Gotchas & honest limitations

The things that trip people up — and the caveats you should volunteer before a colleague discovers them.

Why serve imports the SLIMS stack lazily

cli.py imports requests/slims inside the snapshot/dump commands, not at module load — so serve (which only reads the local snapshot) never pulls in the network/SLIMS stack at all. Keeps the browse path light and dependency-decoupled from the extract path.

Sparse legacy records

The oldest mice have empty mammoid / generation / experiment number — the populated experimental cohort is at higher primary keys. So a small --limit snapshot (which grabs the oldest first) looks emptier than reality. Use a full snapshot to judge coverage.

Two “generation” columns

cntn_cf_generationMm (better populated, P… values) vs cntn_cf_generation (the G… series like G7(5)). We keep both; the model exposes generation and generation_mm.

dna_sequenced is always false

The DNA content type exists but has zero records today, so this derived flag is always false. It's shown but flagged — confirm with the lab whether DNA is tracked under another type before trusting it.

09 Now explain it to a colleague

The real test. Try answering each out loud before opening the model answer.

“Walk me through what happens when I run just snapshot.”

cli parses the snapshot subcommand → snapshot.run() loads .env creds (client.load_config), opens a SLIMS connection (client.connect), and pages every tumor/mouse/treatment/assay content type out of the Content table (extract.fetch_all). Each record is reshaped into a pydantic model (transform.to_tumor etc.), treatments are resolved through an FK lookup, and tumors get their derived fields via the mammoid index. Everything is written to a fresh SQLite file with a full-text index (store.write_snapshot). All of this runs behind the VPN; the web app never repeats it.

“Why is the web app so simple — no React, no API?”

It's a single-user, local, read-only data browser over a static snapshot. Server-rendered Jinja templates plus HTMX give live filtering with no build step, no Node, and no separate API layer. Flask was already available transitively. The complexity budget went into the data pipeline, not the frontend.

“Can I trust the mouse→tumor links?”

Partially. Linking is by the mammoid field, which is clean on tumors but free text on mice, so only ~14% of mice match a tumor exactly — and most of the rest are cell-line models with no patient tumor at all. The drill-down shows matched links honestly and says when nothing matched. There's no true foreign key; eartags + tissue samples could bridge more PDX links if we needed higher coverage.

“Where's the data and how fresh is it?”

A SQLite file at .omnix/snapshot.db, produced by the last omnix snapshot run. Freshness = whenever that was; snapshot_meta records the timestamp and source URL. Re-run just snapshot to refresh. The web app never talks to SLIMS itself.
